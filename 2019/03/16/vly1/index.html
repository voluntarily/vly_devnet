Voluntari.ly 0.0.1

2019-03-16

Tēnā Koutou,
The initial code base for the Voluntari.ly platform is now open for contributors. If you are interested in doing some development work on the platform you are now encouraged to download and work on the vly1 code base.
git clone https://github.com/voluntarily/vly1

So far all the work has been done by myself (Andrew) and Walter. So even a couple of extra hands will make a huge difference.
If you are interested in helping add your name to the CONTRIBUTORS file and submit a pull request.

or

or just email me at andrew@voluntari.ly

Quickstart

  • npm run start - starts the development server with hot reloading enabled
  • npm run test - start the test runner

MongoDB

You’ll need to install mongodb locally or edit the config to point at another server.
For MongoDB installation guide see this. Also npm6 is required to install dependencies properly.

MERN Stack

The Vly1 codebase was created using the Hashnode Mern starter. https://github.com/zacharystenger/mern-starter-JWT-auth

Most of that is unchanged but we have adopted a folder structure model that puts all the files required for a module or component into one subfolder - the model, views, controllers etc. Rather than keeping all the models in one place etc. So everything to do with client side use of Organisations will be in the client/modules/org folder, while all the server side equivalents will be in the server/api/organisation folder.

Whats in the app?

Currently almost nothing. This is little more than a hello world app.

We have:

  • basic entities in the API: activity, opportunity, organisation, person.
  • simple CRUD front end pages for Person and Opportunity.
  • some basic components; Button, ActivityCard, CategoryCard
  • A /showcase page where you can see what the CSS looks like and can see the current component set.
  • Some test coverage

The purpose of these bits is that they set out the basic style of how we can construct the system.

Design

Walter has been working hard on the page design and user experience workflows. These will undergo paper tests with teachers and volunteers over the next few weeks but there is enough in place to get started.

What is the Road Map?

Our MVP is a really simple application that does the following:

  • People with volunteering opportunities can sign up and post an offer.
  • Everyone can browse the offers
  • People who want to volunteer can sign up and express interest in the opportunity.
  • we put people in contact with the organisers and they manage everything after that.
  • there are some informational pages about voluntari.ly

Clearly not the full vision for Voluntari.ly and only two sides of our VOA* triangle. but this is something that we will be able to use straight away. Thats our goal, that the platform however simple is useful for something from the start.

*VOA - Volunteers, Opportunities and Activities.

To get there we need simply to create the Opportunity API, listings page, people sign up and login and some static informational pages.
Then setup a cloud server and deploy to production.

We want to have this in place by TechWeek in May when we will have a Hackathon that can start building features on the base.

Miscellaneous

Deployment server

We have not yet set up any test or operational servers. Thats still todo - any dev/ops contributors want to help with that we would be grateful.

Forms

We have not yet decided on the best way to do Forms. Voluntari.ly is necessarily heavy on forms to create people, orgs, activities and opportuntites so we are looking for a way to minimise code writing to get good validated and flexible forms.

Docker

There is a dockerfile but I’ve not run it yet. If someone tries this let me know if its good or there are any problems.

Redux

Note that the codebase is React+Redux - that means all the client side state is help in a core object and actions are used to track events such as requests for data from the API and changes on pages.
